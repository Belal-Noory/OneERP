=============================
OneERP Planning Docs (v0.1)
=============================

Purpose
- This document is the single source of truth for planning OneERP as a modular, multi-tenant SaaS ERP platform.
- No code should be written unless it follows these rules.


1) Product Definition
1.1 What We Are Building
- A modular ERP platform where each customer company (tenant) can enable/disable modules based on subscription.
- The core platform provides identity, tenant isolation, billing/subscription gates, module registry, navigation, localization, reporting standards, and governance.

1.2 What We Are NOT Building (at the beginning)
- Not a bundle of unrelated CRUD apps.
- Not microservices-first.
- Not “all modules at once”.
- Not a marketplace on day one.


2) Glossary (Canonical Terms)
- Tenant: A company/organization that owns isolated data and configuration (branding, language defaults, modules enabled).
- User: A person account that belongs to one or more tenants (initially: assume one tenant per user for MVP unless explicitly required).
- Membership: The link between a user and a tenant, including role assignment.
- Role: A named set of permissions scoped to a tenant.
- Permission: A capability (e.g., shop.orders.read) that gates UI + API access.
- Module: A packaged feature area (Shop, Pharmacy, Fuel) with its own routes, permissions, navigation items, and reporting definitions.
- Subscription: What a tenant has paid for (plan + module add-ons) which controls module access and limits.


3) Non-Negotiables (Platform Laws)
3.1 Multi-Tenant Isolation
- Every request must resolve the tenant context.
- Every persisted row in tenant-owned tables must include tenant_id.
- Every query for tenant-owned data must filter by tenant_id.

3.2 Access Control (RBAC + Subscription Gates)
- A user must pass BOTH:
  - Subscription gate (tenant has the module/feature enabled)
  - Permission gate (user role includes required permission)
- UI visibility must be derived from the same gates (no “UI shows it but API blocks it” mismatch).

3.3 Localization (EN/LTR + FA/RTL + PS/RTL)
- No hardcoded user-facing text (UI labels, report titles, export headers, errors, email templates).
- All text uses translation keys.
- RTL flips layout direction (sidebar moves to the right, alignment changes, icon placement respects direction).

3.4 Reporting & Exports
- Every data table must support Export: Excel and PDF.
- Exports must respect current filters/search/sort.
- Exports must be language-aware and tenant-branded (logo + details).

3.5 Consistent Layout and Components
- Modules must not introduce custom layout structures.
- Pages follow one page structure (title, actions, filters, data, pagination).
- Shared component library is used everywhere; no “module-specific button styles”.


4) Core Platform Scope (MVP)
4.1 Core Services (Conceptual)
- Auth & Sessions: login, refresh, password reset, MFA-ready later.
- Tenant Management: create tenant, tenant settings, branding, default language.
- User & Membership: invite users, manage memberships, role assignment.
- RBAC: roles, permissions, enforcement.
- Module Registry: list modules, module metadata, enabled state per tenant.
- Subscription/Billing: plan + module add-ons, trial rules, access gates.
- Localization: language selection per user, RTL support rules.
- Reporting Framework: standard export behavior and templates.
- Audit Log: track sensitive events (login, role changes, exports, deletions).

4.2 Admin Surfaces (Conceptual)
- Public website: marketing pages + module catalog + pricing + login/register.
- SaaS app: onboarding + tenant dashboard + module navigation + settings.
- Internal admin (later): manage tenants, modules, subscriptions, support tooling.


5) Canonical User Flows
5.1 Acquisition and Signup
- User visits public website.
- User registers.
- User creates tenant (company) + chooses language default.
- User enters onboarding wizard.

5.2 Onboarding Wizard (MVP)
- Step 1: Company details (name, address, phone, email), upload logo.
- Step 2: Choose business type (used only to suggest modules, not to lock anything).
- Step 3: Select modules (trial-enabled or paid-enabled depending on plan).
- Step 4: Create first admin role and invite initial team members (optional for MVP).
- Step 5: Land on dashboard.

5.3 Daily Use
- Sidebar shows only enabled modules + items permitted for the user.
- Within a module, sub-navigation follows module manifest.
- Tables provide export actions (Excel/PDF).


6) Module System Contract (Definition)
6.1 Module Manifest (Minimum Metadata)
Each module must define a manifest (conceptual structure):
{
  "id": "shop",
  "name_key": "module.shop.name",
  "icon": "shopping-cart",
  "version": "1.0.0",
  "nav": [
    { "label_key": "shop.nav.overview", "path": "/shop/overview", "permission": "shop.overview.read" },
    { "label_key": "shop.nav.orders", "path": "/shop/orders", "permission": "shop.orders.read" }
  ],
  "permissions": [
    "shop.overview.read",
    "shop.orders.read",
    "shop.orders.create",
    "shop.orders.update",
    "shop.orders.delete",
    "shop.reports.export"
  ],
  "reports": [
    { "id": "orders", "title_key": "report.shop.orders.title" }
  ]
}

6.2 Module Rules
- A module cannot bypass the platform’s tenant context.
- A module cannot ship UI navigation without declaring permissions.
- A module cannot add exports without using the platform reporting standard.
- A module cannot introduce new design styles outside the design system tokens.

6.3 Module Enablement
- Modules are enabled per tenant.
- Enablement is controlled by subscription state and plan rules.
- If a module is disabled, all module routes and APIs are inaccessible.


7) Authorization Model (RBAC) - Planning Rules
7.1 Permission Naming
- Format: {module}.{resource}.{action}
- Examples:
  - shop.orders.read
  - shop.orders.create
  - pharmacy.inventory.update
  - platform.users.invite

7.2 Default Roles (MVP)
- Owner (tenant-level super admin): all permissions, billing access.
- Admin: manage users/roles/modules (except billing optional).
- Manager: operational permissions for enabled modules.
- Staff: limited read/write within module scope.
- ReadOnly: read-only + export optional depending on policy.

7.3 Permission Enforcement
- Every API endpoint declares required permission(s).
- UI routes and navigation items declare required permission(s).
- “Export” actions are gated by explicit export permission.

    "trial_days": 14
  }
}

16.3 Validation Rules
- id: lowercase, kebab-case, unique across platform (e.g., shop, fuel-management).
- version: semantic version (major.minor.patch).
- name_key/label_key/title_key: required and must exist for EN/FA/PS.
- entry_path must match one of nav paths.
- nav.path must be unique within module and must start with "/{moduleId}".
- Every nav item must reference exactly one permission key declared in permissions.
- Every API route must declare permissions (permissions_any or permissions_all).
- Every report must declare permission_export for export buttons.
- No permission key may contain spaces or uppercase letters.

16.4 Module Lifecycle (Platform Behavior)
- Install/Enable:
  - Adds module to tenant enabled modules set if subscription allows.
  - Sidebar/nav becomes visible only after enable and permission checks.
- Disable:
  - UI navigation removed.
  - API access blocked regardless of user permissions.
  - Existing data remains in database unless a dedicated data purge action exists (not MVP).
- Upgrade/Downgrade:
  - Manifest version changes must be backward compatible unless major version is explicitly migrated.

20) API Standards (Platform-Wide Contract)
20.1 Goals
- All modules expose APIs consistently.
- Clients can rely on predictable pagination, filtering, sorting, and error shapes.
- RBAC + subscription gates are enforced consistently.

20.2 API Naming Conventions
- REST-style resource endpoints.
- Use plural nouns: /api/shop/orders
- Nested resources only when it improves clarity: /api/shop/orders/{orderId}/items
- Use kebab-case for URL segments only if strictly needed; prefer simple lowercase.

20.3 Required Request Context
- Authentication required for all /api/* endpoints except public auth endpoints.
- Every tenant-owned endpoint requires tenant context:
  - X-Tenant-Id must be present and validated against membership.
- Localization context:
  - The user’s profile language is primary.
  - Client may pass Accept-Language for UI-driven calls, but server must validate supported locales.

20.4 Standard Response Shape
Success (single)
{
  "data": { ... },
  "meta": { }
}

Success (list)
{
  "data": [ ... ],
  "meta": {
    "page": 1,
    "pageSize": 25,
    "total": 123,
    "sort": ["createdAt:desc"],
    "filters": { }
  }
}

Error
{
  "error": {
    "code": "PERMISSION_DENIED",
    "message_key": "error.permissionDenied",
    "details": { }
  }
}

20.5 Error Codes (Minimum Set)
- UNAUTHENTICATED
- TENANT_REQUIRED
- TENANT_ACCESS_DENIED
- SUBSCRIPTION_REQUIRED
- MODULE_DISABLED
- PERMISSION_DENIED
- VALIDATION_ERROR
- NOT_FOUND
- CONFLICT
- RATE_LIMITED
- INTERNAL_ERROR

20.6 Validation Standards
- All create/update endpoints validate input strictly.
- Validation errors return VALIDATION_ERROR with field-level errors inside details.
- No endpoint may accept unknown fields silently.

20.7 Pagination
- Query parameters:
  - page (default 1)
  - pageSize (default 25, max 200)
- Must return total for all list endpoints in MVP.

20.8 Sorting
- Query parameter:
  - sort=field:direction
  - Multiple: sort=createdAt:desc,name:asc
- Only allow sorting on a whitelist per endpoint.

20.9 Filtering
- Use query parameters with explicit operators:
  - equals: field=value
  - contains: field__contains=value
  - range: field__gte=, field__lte=
  - enum: status=in(draft,completed)
- Each endpoint must publish supported filters (derived from module manifest or endpoint contract).

20.10 Idempotency
- For endpoints that create financial or irreversible operations (orders, payments, exports optional):
  - Support Idempotency-Key header to prevent duplicates.

20.11 Export Endpoints (Standard)
- Every report/table must support:
  - POST /api/{module}/reports/{reportId}/export/pdf
  - POST /api/{module}/reports/{reportId}/export/excel
Request body includes:
- filters
- sort
- columns (optional override)
- dateRange (optional)
Rules
- Server-generated exports preferred for consistent fonts and RTL correctness.
- Export response options:
  - Immediate download (file stream) for small exports
  - Async job + polling for large exports (future; define when needed)

20.12 Auditing Requirements (API)
- The following must create audit events:
  - user invites, role changes, permission changes
  - module enable/disable
  - subscription changes
  - export actions
  - destructive actions


21) Subscription States, Limits, and Gating Rules (MVP)
21.1 Subscription Status Behavior
- trialing:
  - Access allowed to selected modules under trial rules until trial_ends_at.
- active:
  - Full access based on plan + module add-ons.
- past_due:
  - Access allowed during grace period (grace_ends_at).
- suspended:
  - Block module usage; allow login + billing only.
- canceled:
  - Block module usage; allow export of billing history; data remains.

21.2 Plan Limits (Conceptual)
- Each plan can set limits:
  - max_users
  - max_modules_enabled
  - max_storage_mb
  - exports_per_day (optional)
Rules
- Limits must be enforced server-side.
- UI may show warnings but server is the source of truth.

21.3 Module Add-On Rules
- A module is usable only if:
  - tenant_enabled_modules.status = enabled
  - subscription allows module_id (add-on active or plan includes it)
- Disabling an add-on must disable module access immediately (except explicitly defined grace rules).


22) Security & Compliance Baseline (MVP)
22.1 Goals
- Prevent tenant data leaks.
- Protect accounts from common attacks.
- Provide auditability for enterprise customers.
- Keep security rules consistent across all modules.

22.2 Authentication Security
Passwords
- Store only password hashes (never plaintext, never reversible encryption).
- Use a modern password hashing algorithm appropriate for production (configurable work factor).
- Enforce minimum password length and block common passwords.

Sessions/Tokens
- Short-lived access token + longer-lived refresh token.
- Refresh tokens must be revocable per user session.
- Logout revokes refresh token.
- Rotate refresh tokens on use where possible.

Account Recovery
- Password reset links/tokens must be time-limited and single-use.
- Do not reveal whether an email exists in the system during reset flows.

MFA Readiness (Not necessarily MVP)
- Plan for MFA enrollment and challenge without breaking existing auth flows.

22.3 Authorization Security (RBAC + Subscription)
- Every API endpoint must enforce:
  - tenant membership
  - subscription gate
  - permission gate
- Never rely on UI-only checks for security.
- Deny-by-default: if permission is not declared, access is blocked.

22.4 Tenant Isolation Guarantees
- Tenant context must be resolved before any query.
- Tenant-owned tables must always filter by tenant_id in queries.
- No cross-tenant joins unless in internal admin context with explicit safeguards.
- Avoid “admin override” inside the main SaaS app; keep it in internal admin tooling.

22.5 Input Validation & Output Safety
- Validate inputs on all create/update endpoints.
- Reject unknown fields (no silent acceptance).
- Use consistent validation error structure (see API Standards).
- Protect against injection by using parameterized queries and safe query builders.

22.6 Rate Limiting & Abuse Prevention (MVP Minimum)
- Apply rate limiting to:
  - login
  - password reset request
  - invite user
  - export endpoints (to prevent abuse)
- Add basic bot protection to public registration (captcha optional, depends on region and UX).

22.7 Sensitive Data Handling (PII)
- Identify PII fields (email, phone, address, names).
- Do not log PII in application logs unless strictly required.
- Redact secrets, tokens, and password reset codes from logs.

22.8 Audit Log Policy (Security-Relevant)
- Audit must include:
  - login failures
  - role and permission changes
  - module enable/disable
  - subscription changes
  - export events
  - destructive actions
- Audit events must include:
  - actor_user_id
  - tenant_id
  - timestamp
  - entity_type/entity_id when applicable

22.9 Data Protection (At Rest / In Transit)
- Enforce HTTPS everywhere in production.
- Encrypt sensitive secrets in environment configuration.
- File storage must support access control by tenant_id.
- Export files (if stored) must not be publicly accessible by URL without authorization.

22.10 Security Operations (MVP Practical)
- Provide a way to:
  - deactivate users
  - revoke sessions
  - rotate secrets (server-side)
- Track security incidents via audit and logs.


23) Reporting Export Job Strategy (PDF/XLSX)
23.1 Goals
- Exports must be consistent, language-aware, tenant-branded, and safe for large datasets.
- Support both small “instant” exports and large “async” exports.

23.2 Export Modes
Synchronous Export (MVP default)
- Used when result size is within safe limits.
- User clicks export and receives file immediately.

Asynchronous Export (Planned, required for scale)
- Used when result size is large or export generation is slow.
- Export creates a job and returns a job id.
- User can track status and download when ready.

23.3 Export Size Limits (Policy)
- Define safe limits:
  - max_rows_sync_pdf
  - max_rows_sync_excel
  - max_rows_total_export (hard limit or admin policy)
- If request exceeds sync limits:
  - Either switch to async job
  - Or require narrower filters

23.4 Export Job Data Model (Tenant-Owned)
exports (tenant-owned)
- id (uuid)
- tenant_id
- requested_by_user_id
- module_id
- report_id
- format (pdf, excel)
- status (queued, running, completed, failed, canceled)
- request_payload_json (filters/sort/columns)
- result_file_id (fk files.id, nullable until completed)
- rows_exported (nullable)
- error_code (nullable)
- error_message_key (nullable)
- created_at, started_at, completed_at

23.5 Export Security Rules
- Export permissions:
  - Must require explicit export permission (module-specific or platform export).
- Export contents:
  - Must apply the same authorization filters as the UI list.
  - Must not bypass tenant_id enforcement.
- Download access:
  - Only users with membership in tenant and export permission can download.
  - Signed URLs must be short-lived if used.

23.6 Export Localization Rules (Reinforced)
- Exports use the user’s current selected language at export time.
- PDF direction:
  - EN: LTR
  - FA/PS: RTL
- Fonts used in exports must support FA/PS characters; avoid missing glyphs.

23.7 Tenant Branding Rules (Reinforced)
- PDF header must show:
  - tenant logo if present, otherwise OneERP logo
  - tenant name and contact details

23.8 Export Retention Policy (MVP)
- Default retention: store generated exports for a limited period (e.g., 7–30 days) then delete.
- Allow tenant policy later (enterprise tier) to configure retention.
- Audit must record export events even if files are later deleted.

23.9 Failure Handling
- If an export fails:
  - show a localized error message using message_key
  - record failure in audit log with error_code
  - do not leak internal stack traces to the user


24) Observability & Operations (MVP Baseline)
24.1 Goals
- Debug production issues quickly without exposing sensitive data.
- Detect tenant-impacting incidents early (exports failing, login spikes, DB errors).
- Provide enough signals to scale later (jobs, modules, subscription gates).

  - job logs (exports)
  - audit events (metadata_json) when relevant

- Minimum log fields:

  - level
  - requestId
  - tenantId (if available)
  - userId (if available)
  - moduleId (if applicable)
  - action or eventName
  - durationMs (for requests)
  - errorCode (for failures)
- Never log:
  - passwords
  - tokens
  - password reset codes
  - full PII payloads

24.5 Alerts (MVP Minimum)

  - export failure rate above threshold

24.6 Health Checks
- Provide health endpoints:
  - liveness (process up)
  - readiness (db reachable, critical dependencies reachable)

24.7 Audit vs Logs
- Logs are for debugging and are not guaranteed permanent.
- Audit is for compliance and must be tenant-scoped and immutable.

  - typography sizes/weights
  - spacing
  - radii
  - shadows
- No inline styles.
- No one-off component variants outside the shared component library.

27.3 Layout Patterns (Non-Negotiable)
- Global app layout:
  - sidebar + header + content area
- Page layout:
  - title
  - primary actions
  - filters/search
  - content (table/cards)
  - pagination

27.4 Table Pattern (Standard)
- All list pages use one table component with:
  - column definitions
  - sorting
  - filtering/search
  - pagination
  - export dropdown
- Empty states must be consistent:
  - empty (no data yet)
  - no results (filters returned nothing)
  - error (failed to load)

27.5 Forms Pattern (Standard)
- All forms use one form system with:
  - labels, helper text, errors
  - consistent field spacing and widths
  - localized validation messages (message_key)

27.6 RTL UI Rules (Reinforced)
- RTL affects:
  - sidebar position (right)
  - padding/margins direction
  - icon placement for directional icons
  - table alignment rules
- Numeric values remain formatted consistently and aligned predictably.

27.7 Accessibility (MVP Minimum)
- Keyboard navigation for:
  - sidebar
  - tables
  - modals
- Visible focus states.
- Text contrast must meet basic readability standards.


28) Versioning, Compatibility, and Rollout Rules
28.1 Goals
- Avoid breaking tenants during updates.
- Keep modules upgradeable without corrupting data.
- Make changes predictable for support and QA.

28.2 Versioning
- Platform version: semantic versioning.
- Module manifest version: semantic versioning.
- Breaking changes require major version bump and a migration plan.

28.3 Backward Compatibility Rules
- API changes must be backward compatible within a minor version.
- Deprecations require:
  - a deprecation window
  - clear migration path
- Database migrations must not break running code during rolling deploys:

  - avoid dropping columns until after code is migrated

28.4 Feature Flags

  - billing enforcement changes
  - module enablement UX changes

28.5 Data Migration Strategy (When Needed)

  - provide rollback strategy where possible

32.4 Registration Flow (High-Level)
Step A: Create account
- User provides identity fields.
- Account is created (users table).

Step B: Create tenant (company)

- Tenant is created (tenants + tenant_branding baseline).

Step C: Create membership
- Membership created for the user as Owner in that tenant (memberships).
- Owner role exists (system role) or is created automatically for the tenant.

Step D: Redirect into SaaS app
- User is authenticated and redirected to:
  - /t/{tenantSlug}/dashboard (or first onboarding page)

32.5 Registration Form Requirements (Fields + Validation)
Account Fields (MVP)
- full_name (required, min length 2)
- email (required unless phone auth is enabled, must be valid email)
- password (required, min length policy from section 22.2)
- language preference (optional; default to browser locale if supported, else tenant default later)

Company (Tenant) Fields (MVP)
- company_legal_name (required)
- company_display_name (required; can be same as legal)
- tenant_slug (required, unique; may be auto-generated from display name with user override)
- default_locale (required: en/fa/ps)
- address (optional in MVP but recommended)
- phone (optional)
- email (optional)
- logo upload (optional in MVP; required later for premium exports)

Validation Rules
- tenant_slug:
  - lowercase, alphanumeric + hyphen only
  - length limits (e.g., 3–50)
  - must be unique across all tenants
- password:
  - follow password policy in section 22.2
- email/phone:
  - do not allow duplicates across users if they are used as login identifiers

32.6 Required Backend Side Effects (What Must Be Created)
On successful registration + tenant creation:
- users row created (global)
- tenants row created (tenant root)
- tenant_branding row created (baseline)
- tenant_settings row created (baseline)
- roles: Owner and Admin defaults created for tenant (or referenced as system templates)
- memberships row created linking user to tenant with Owner role
- subscriptions row created for tenant:
  - status = trialing or active depending on business decision (section 21)
  - trial_ends_at set if trialing
- tenant_enabled_modules:
  - optionally enable a default “core” module set (dashboard/settings) that is not billable

32.7 Authentication Entry Points
Login
- Accept email + password (MVP).
- After login:
  - if user has one membership: redirect to that tenant
  - if user has multiple memberships: show tenant picker then redirect

Logout
- Revokes refresh token (section 22.2).

Password Reset
- Request reset without revealing whether email exists (section 22.2).

- Rate limit:
  - registration
  - login
  - password reset

- Prevent account enumeration in auth flows.
